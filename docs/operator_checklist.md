# Операторский чеклист

Документ для ситуации “бот не отвечает”, “админка странная”, “материалы не ищутся” или “завтра запуск, надо проверить”.

## Главное правило

Не делать разрушительных действий без backup.

Не выполнять:

```bash
docker compose down -v
```

Не удалять Docker volumes.

## Быстрая проверка локально

В папке проекта:

```powershell
docker compose ps
```

Нормально:

```text
leader_bot_app        Up
leader_bot_postgres   Up ... healthy
```

Логи:

```powershell
docker compose logs --tail=100 bot
```

Нормально:

```text
Bot started in polling mode
Run polling for bot @...
```

## Быстрая проверка на сервере

Подключиться:

```bash
ssh ubuntu@SERVER_PUBLIC_IP
cd ~/apps/LL-bot
docker compose ps
docker compose logs --tail=100 bot
```

## Если бот не отвечает в Telegram

Проверить по порядку:

1. Контейнер `bot` запущен.
2. Контейнер `postgres` healthy.
3. В логах нет постоянных ошибок.
4. Нет второй копии бота с тем же token.
5. В `.env` правильный `TELEGRAM_BOT_TOKEN`.
6. Telegram API доступен с машины.

Команды:

```bash
docker compose ps
docker compose logs -f bot
```

Если локально и на сервере запущены два бота с одним token, один выключить.

Локально:

```powershell
docker compose down
```

На сервере:

```bash
docker compose stop bot
```

## Если админка не открывается

Проверить:

- пользователь отправляет `/admin`;
- Telegram ID пользователя есть в `TELEGRAM_ADMIN_IDS`;
- после изменения `.env` контейнер бота перезапущен.

Перезапуск:

```bash
docker compose restart bot
```

## Если OpenAI не отвечает

Симптомы:
- бот пишет “Сейчас не получилось получить ответ от модели”;
- `Админ: статус` показывает ошибку LLM chat или embeddings.

Проверить:

- `OPENAI_API_KEY` заполнен;
- ключ не отозван;
- есть баланс/лимиты;
- модель в `.env` существует;
- сервер имеет доступ в интернет.

Команда логов:

```bash
docker compose logs --tail=200 bot
```

Что можно сделать безопасно:

```bash
docker compose restart bot
```

## Если база не healthy

Симптомы:
- `leader_bot_postgres` не healthy;
- бот падает при старте;
- ошибки подключения к PostgreSQL.

Проверить:

```bash
docker compose ps
docker compose logs --tail=100 postgres
```

Частые причины:
- неверный пароль Postgres в `.env`;
- занятый порт 5432;
- повреждён volume;
- нехватка места на диске.

Проверить диск:

```bash
df -h
```

Не удалять volume без backup.

## Если файл не загружается

Проверить:

- формат: PDF, DOCX, PPTX, TXT;
- размер меньше лимита;
- файл не повреждён;
- Telegram смог скачать файл;
- OpenAI embeddings доступны.

Админ может нажать:

```text
Админ: статус
```

## Если материал загружен, но бот его не использует

Проверить:

- материал есть в `Админ: материалы`;
- статус `ready`;
- вопрос действительно связан с материалом;
- материал содержит извлекаемый текст.

Для PDF:
- сканы без текстового слоя могут плохо извлекаться;
- нужен текстовый PDF или OCR, которого в MVP пока нет.

Если всё выглядит нормально, но поиск плохой:

```text
/reindex
```

Использовать `/reindex` аккуратно, потому что он тратит embeddings.

## Если ответы слишком общие

Пока не менять код вслепую.

Собрать:
- пример вопроса;
- ответ бота;
- источники в ответе;
- оценку `Не полезно`;
- причину `Слишком общий` или `Не нашёл ответ`.

После 5-10 примеров можно точечно править prompt или материалы.

## Если кнопка не нажимается

Проверить:

- бот не ждёт файл в текущем состоянии;
- нажата актуальная кнопка из текущего меню;
- пользователь не пишет старую кнопку вручную;
- контейнер `bot` работает.

Универсальный сброс:

```text
/cancel
```

Потом:

```text
/start
```

## Если админ менял текст и ошибся

Если текст ещё на preview:

```text
Отменить
```

Если уже сохранил:

- снова открыть `Админ: тексты`;
- выбрать нужный блок;
- отправить исправленный текст;
- сохранить.

Удаления текстов нет, это специально безопаснее для MVP.

## Перед занятием

За 30-60 минут:

1. Открыть `/admin`.
2. Нажать `Админ: статус`.
3. Проверить БД OK.
4. Проверить LLM chat OK.
5. Проверить embeddings OK.
6. Нажать `Админ: материалы`.
7. Проверить нужные материалы.
8. Задать тестовый вопрос как обычный пользователь.
9. Проверить, что ответ приходит.
10. Не запускать локальную копию, если серверная уже работает.

## После занятия

1. Проверить `Админ: статистика`.
2. Посмотреть количество `Не полезно`.
3. Собрать плохие ответы.
4. Не править prompt сразу после одного плохого ответа.
5. Сначала понять, проблема в материале, вопросе, поиске или prompt.

## Минимальный backup перед рискованными действиями

На сервере:

```bash
mkdir -p ~/backups
docker compose exec -T postgres pg_dump -U postgres -d leader_bot > ~/backups/leader_bot_$(date +%F_%H-%M).sql
tar -czf ~/backups/leader_bot_data_$(date +%F_%H-%M).tar.gz data
ls -lh ~/backups
```

После backup можно безопаснее делать обновления.
